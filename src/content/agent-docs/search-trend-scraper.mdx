mechanisms:
  - id: reliable-period-window
    claim: Proportional search trend uses a fixed historical proportion window (2023-01-01 to 2025-04-30) rather than rolling weekly proportions.
    evidence: utils/analysis.py:13-35
    rationale: Review data after 2025-04-30 is incomplete due to site-level blocking of the review source. Using a fixed window avoids biasing recent search trend estimates with sparse review data.
  - id: naver-datalab-batch-sleep
    claim: Naver Datalab batches brands in groups of 5 with a 2-second inter-call sleep and 5-second batch sleep.
    evidence: scraper.py:199-230
    rationale: Naver Datalab API limits keyword groups to 5 per request. Sleeps exist to avoid rate-limit errors across 800+ brands (~1-2 hour total runtime).
  - id: csv-checkpoint
    claim: CSV checkpointing every 100 brands in scraper.py.
    invariant: Partial progress is persisted mid-run; main.py auto-selects *_final.csv or falls back to the largest file.
    evidence: scraper.py:233-236, main.py:31-37
    rationale: Guards against data loss in the event of a mid-run network or API failure.
  - id: idempotent-drop-before-tosql
    claim: create_versioned_table always begins with DROP TABLE IF EXISTS before df.to_sql.
    invariant: A previous failed attempt that partially created the table is silently cleared on retry.
    evidence: utils/tbl_version_manager.py:48-49
    rationale: Provides idempotency on retried runs.
  - id: split-transaction-activation
    claim: TableVersionManager records the version entry and deactivates prior versions in separate engine.begin() transactions.
    evidence: utils/tbl_version_manager.py:65-87
    rationale: "No explicit rationale in code. Consequence: if the INSERT into table_versions succeeds but is_active deactivation fails (or vice versa), the metadata registry can have two active rows for the same table type."
