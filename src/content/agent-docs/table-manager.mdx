mechanisms:
  - id: transactional-activation
    claim: activate_version wraps deactivate-all then activate-one in a single begin() transaction.
    invariant: Exactly one version is active per table type; the transaction boundary enforces it without triggers or constraints.
    evidence: src/database/tbl_version_manager.py:106-119
    rationale: No moment exists where zero or two versions are active; a partial write cannot violate the invariant.
  - id: db-side-version-allocation
    claim: The next version number is allocated inside SQL via COALESCE(MAX(version), 0) + 1.
    invariant: Race-safe version numbering when multiple pipeline runs attempt creation simultaneously.
    evidence: src/database/tbl_version_manager.py:27
    rationale: The database serializes the MAX() read rather than relying on Python-side coordination.
  - id: schema-cloning
    claim: New versioned tables are created via CREATE TABLE new (LIKE source INCLUDING ALL).
    invariant: All indexes, constraints, and column types are preserved exactly from the source table.
    evidence: src/database/tbl_version_manager.py:94
    rationale: Higher fidelity than SELECT INTO or pandas to_sql type inference. A DataFrame-only path exists as a fallback when no source table is provided.
  - id: dry-run-first-class
    claim: --dry-run is a first-class flag on --cleanup, routed through the same cleanup_old_versions method via a dry_run boolean parameter.
    evidence: src/database/tbl_version_manager.py:165-197
    rationale: Single code path, no duplicated logic. Visual distinction is a Rich panel style shift (dim in dry-run mode).
  - id: chunked-bulk-insert
    claim: Bulk inserts use pandas to_sql in 500-row chunks with a Rich progress bar (SpinnerColumn + BarColumn + MofNCompleteColumn).
    invariant: Memory-safe loads of large DataFrames with live row-level progress feedback.
    evidence: src/database/tbl_version_manager.py:54-70
  - id: cleanup-active-protection
    claim: cleanup_old_versions filters WHERE is_active = FALSE before applying OFFSET :keep in its candidate-selection SQL.
    invariant: The active version is never in the delete candidate set, regardless of how old it is.
    evidence: src/database/tbl_version_manager.py:168-176
