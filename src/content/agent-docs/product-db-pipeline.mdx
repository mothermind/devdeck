dependencies:
  - name: MetadataPreprocessor
    path: src/preprocessing/metadata_preprocessor_async.py
    kind: module
    role: Chains GPT name cleaning (ProductNameCleaner), BERT embeddings, DBSCAN clustering, and fuzzy matching to group similar products and select representatives; called by Step 2.
  - name: AsyncOCRExtractor
    path: src/feature_extraction/ocr_extractor_async.py
    kind: module
    role: Runs Google Cloud Vision OCR on S3 product images using ThreadPoolExecutor(max_workers=5); skips items with existing DynamoDB keywords >= 35 chars; called by Step 3.
  - name: KeywordExtractor
    path: src/feature_extraction/keyword_extractor.py
    kind: module
    role: Loads category-specific questions/options from PostgreSQL, submits OpenAI Batch API jobs, manages session IDs via BatchJobManager, and processes results with option matching; called by Step 5.
  - name: ImagePipeline
    path: src/database/db_connector.py
    kind: module
    role: Fetches S3 images as base64 and retrieves OCR keywords from DynamoDB for analysis-ready data preparation in Step 4.
  - name: ProductAnalysisProcessor
    path: pipelines/product_analysis_processor.py
    kind: module
    role: Processes new AI analysis results row-by-row and concatenates them with existing product_analysis data for the versioned table update in Step 6.
  - name: FeatureEngineeringPipeline
    path: pipelines/feature_engineering_pipeline.py
    kind: module
    role: Computes item-level features (popularity_index, lifecycle_stage, vitality_score, price/size/discount features) and group-level aggregates written to product_service in Step 7.
  - name: TableVersionManager
    path: src/database/tbl_version_manager.py
    kind: module
    role: All versioned PostgreSQL table creation, activation, and cleanup lifecycle; documented separately as the table-manager agent-docs entry — not re-documented here.
  - name: product_raw
    kind: postgres-table
    role: Versioned table (product_raw_{version}) written by Step 2; contains deduplicated representative products with group assignments; consumed by Steps 4 and 6 and downstream scrapers.
  - name: product_analysis
    kind: postgres-table
    role: Versioned table (product_analysis_{version}) written by Step 6; contains AI-extracted feature answers per product option; consumed by Step 7 and downstream analytics.
  - name: product_service
    kind: postgres-table
    role: Versioned table (product_service_{version}) written by Step 7; contains fully engineered features including popularity_index, lifecycle_stage, and vitality_score; final output for downstream consumers.
  - name: OliveYoung_ItemKeywords / BK_ItemKeywords
    kind: service
    role: DynamoDB tables storing OCR-extracted keyword strings per product image; written by Step 3 and read by Step 4.

mechanisms:
  - id: step1-parallel-dynamodb-scan
    claim: Step 1 performs a parallel 20-segment DynamoDB scan of OliveYoung_MetaData and BK_MetaData, standardizes date formats to ISO 8601, maps BK brand IDs, filters by category_source table, and deduplicates on itemId.
    evidence: run_steps.py:87-180
  - id: step2-hybrid-grouping
    claim: Step 2 chains GPT name cleaning, BERT-base-multilingual-cased embeddings, DBSCAN clustering (eps=0.3, cosine metric), and fuzzy matching (threshold 0.9) to group K-Beauty products and select representatives before writing product_raw.
    rationale: K-Beauty product names include size variants, shade codes, and promotional suffixes that fool pure string matching; the hybrid handles semantic and string similarity together.
    evidence: "run_steps.py:185-278, src/preprocessing/metadata_preprocessor_async.py:29-46"
  - id: step3-ocr-skip-threshold
    claim: Step 3 skips items whose existing DynamoDB keyword string is >= 35 chars, avoiding redundant Vision API calls on already-extracted items.
    invariant: Incremental OCR runs only process products with no meaningful prior extraction.
    evidence: src/feature_extraction/ocr_extractor_async.py:116-117
  - id: step4-analysis-data-prep
    claim: Step 4 uses ImagePipeline.fetch_product_data() to fetch S3 images as base64 and OCR keywords from DynamoDB, maps category_id_main, and skips group_ids already present in the active product_analysis table.
    evidence: run_steps.py:373-490
  - id: step5-nonblocking-batch-api
    claim: Step 5 submits analysis of ~28K products to OpenAI Batch API non-blockingly via KeywordExtractor.submit_batches(), returning a session_id; status is polled separately and results downloaded independently, allowing the operator to close their laptop between submit and retrieve.
    rationale: Analysis of ~28K products would take hours in a blocking loop; Batch API lets the operator submit and disconnect.
    evidence: "run_steps.py:495-573, run_steps.py:564-571"
  - id: step6-versioned-analysis-table
    claim: Step 6 instantiates ProductAnalysisProcessor with the active product_analysis table name, calls process_data_with_separate_rows() on new results, concatenates with existing data, and creates a new versioned product_analysis table via TableVersionManager.
    evidence: run_steps.py:813-911
  - id: step7-feature-engineering
    claim: Step 7 calls generate_features() for item-level features then transform_to_product_service() for group-level features; popularity_index is computed as log_normalize(review)*0.3 + (rating/5)*0.25 + log_normalize(review_density)*0.2 + min_max_normalize(discount_pct)*0.1 + min_max_normalize(days_since_reg)*0.15.
    evidence: "run_steps.py:916-1022, pipelines/feature_engineering_pipeline.py:85-200, pipelines/feature_engineering_pipeline.py:190-197"
  - id: csv-checkpointing
    claim: After every step, PipelineRunner calls save_checkpoint() to persist a CSV at results/checkpoint_{step_name}.csv; any step can be re-entered from disk by loading the prior checkpoint without re-running upstream steps.
    invariant: Partial progress survives process failure; re-entry cost is a disk read rather than a full upstream rerun.
    rationale: Long-running API calls (OCR, GPT) make full reruns prohibitively expensive; checkpoints let operators restart after failures.
    evidence: run_steps.py:67-80
  - id: table-version-manager-delegation
    claim: Steps 2, 4, 6, and 7 call version_manager.get_active_table() and version_manager.create_versioned_table() for all versioned PostgreSQL table lifecycle; the PipelineRunner exposes TableVersionManager as a lazy-loaded property.
    rationale: Versioning logic is reusable across pipeline stages and warrants its own testable unit.
    evidence: "run_steps.py:59-63, src/database/tbl_version_manager.py"
  - id: from-db-flag
    claim: The --from-db flag on Steps 3 and 4 switches the data source from checkpoint CSV to the active product_raw PostgreSQL table, allowing operators to continue a pipeline run from the live DB state rather than a potentially stale CSV.
    evidence: "run_steps.py:1157-1160, run_steps.py:303-315"
