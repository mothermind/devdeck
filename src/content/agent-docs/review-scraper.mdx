mechanisms:
  - id: dual-extraction-strategy
    claim: Review extraction routes through Shadow DOM scroll-accumulation for the 2025 OY redesign or falls back to BeautifulSoup numbered-page pagination for the legacy layout, selected per product page at runtime.
    invariant: All products are scraped regardless of which OY front-end version is live.
    evidence: "main.py:307-383 (Shadow DOM extraction), main.py:577-640 (legacy BeautifulSoup path)"
    rationale: OY's 2025 redesign moved reviews into a virtualized Shadow DOM custom element (oy-review-review-in-product); standard CSS selectors and pagination no longer work against it.
  - id: shadow-dom-scroll-accumulation
    claim: Shadow DOM extraction traverses the oy-review-review-in-product custom element recursively via JS evaluate and accumulates reviews through repeated scroll with a date+text[:50] dedup key to handle virtual-list recycling.
    evidence: "main.py:307-383 (extract_reviews_from_shadow_dom JS evaluate), main.py:498-546 (scroll loop + accumulated_reviews dict)"
  - id: dynamodb-cutoff-prefetch
    claim: DynamoDB is queried in bulk before scraping begins to build a latest_dates dict keyed by item_id; each worker skips reviews on or before its product's cutoff date.
    invariant: Already-ingested reviews are never re-written to S3 or re-processed by downstream consumers.
    evidence: "main.py:47-71 (DynamoDBHandler.get_latest_date), main.py:937-954 (bulk prefetch loop)"
    rationale: DynamoDB provides O(1) per-product lookups without a full table scan; prefetching in bulk before the asyncio task queue starts avoids per-product async latency.
  - id: semaphore-bounded-concurrency
    claim: All product scrape tasks are enqueued as asyncio tasks gated by asyncio.Semaphore(MAX_CONCURRENCY=2); each worker sleeps RATE_LIMIT (6 s) plus random jitter before opening a page.
    invariant: At most 2 Playwright pages are open simultaneously.
    evidence: "main.py:989 (semaphore = asyncio.Semaphore(MAX_CONCURRENCY)), main.py:1002-1009 (worker fn)"
    rationale: Code comments cite T3 CPU credit preservation as the reason for the conservative concurrency cap (reduced from 5).
  - id: adaptive-cpu-throttle
    claim: adaptive_throttle() reads CPU load via psutil and extends the per-worker delay by 10 s above 80% CPU or 30 s above 95% CPU; an additional batch_cooldown of 30 s with CPU-recovery extension runs between Parquet flushes.
    evidence: "main.py:218-238 (adaptive_throttle), main.py:242-263 (batch_cooldown)"
  - id: batch-size-200
    claim: BATCH_SIZE is currently 200 (not 500); the code comment explicitly notes the reduction from 500 to save progress more frequently and reduce in-memory accumulation between S3 flushes.
    evidence: main.py:207
  - id: incremental-manifest
    claim: _manifest.json is written to S3 at the same prefix after every 200-product batch rather than once at the end; the manifest includes files[], summary totals, and date_range per file.
    invariant: If the run crashes mid-way, the manifest reflects completed batches and can be used for resumption via --run-id.
    evidence: "main.py:138-151 (save_manifest), main.py:1060-1066 (incremental write), main.py:1135-1155 (final manifest)"
  - id: retry-exponential-backoff
    claim: retry_async wraps page operations with exponential backoff (2^attempt seconds) up to MAX_RETRIES=5.
    evidence: main.py:296-304
  - id: stealth-init
    claim: Playwright context overrides navigator.webdriver, injects window.chrome, sets locale to ko-KR and timezone to Asia/Seoul, and spoofs UA as macOS Chrome 120 via an init script.
    evidence: main.py:278-292
    rationale: Reduces detection probability on the Olive Young site.
  - id: parquet-output-schema
    claim: "Output Parquet schema per row: date, item_id, group_id, rating, review_text — Snappy-compressed via PyArrow; S3 key pattern is data/reviews/raw/{YYMMDD}/scraper_results_{N}_{YYMMDD}.parquet."
    evidence: "main.py:800-806 (DataFrame columns), main.py:112 (compression='snappy'), main.py:105-107 (S3 key construction)"
  - id: s3-resume-check
    claim: In production mode (non-dry-run), get_processed_items_from_s3 scans today's S3 prefix and filters already-present item_ids out of the work queue before tasks are dispatched.
    evidence: main.py:864-884
